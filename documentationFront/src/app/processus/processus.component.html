Un modèle Karaz peut contenir un ou plusieurs processus métier. Dans la plupart des cas l’un des processus associés au modèle constitue le processus principal qui est à l’origine de création de toutes les instances du modèle (instance modèle=objet). Ainsi, le lancement d’une instance de ce processus, crée automatiquement l’objet auquel le processus s’applique. A un modèle métier peuvent être appliqués plusieurs processus. Ils ne peuvent cependant tourner simultanément que s’ils sont de type Non Exclusive. La modélisation du processus métier est le premier travail de conception que le développeur ou l’analyste métier effectue sur Karaz Studio dans le cadre de la production d’un module applicatif

Configuration de processus

Tout processus métier sous Karaz peut être configuré à travers un ensemble de propriétés, dont les plus importantes sont regroupées au niveau de l’onglet « Process » de la fenêtre Propriétés. Ces paramètres de configuration du processus  sont listés dans le tableau ci-après :

ATTRIBUT

ROLE

Name

Nom du processus. Il s’agit du nom qui apparaitra au niveau de la console d’administration (Karaz Back End). Ce nom peut être diffèrent du nom du fichier (process_name.bpmn)

Default View

Précise le nom de la vue qui s’affichera par défaut lors du lancement d’une nouvelle instance du processus.

Is Auto Start

Si activé, cet attribut indiquera à Karaz Workflow Engine qu’à la création d’une instance de ce processus par un utilisateur, ce dernier se verra automatiquement affecté la première user activity du modèle. Si l’attribut n’est pas activé la première user activity du processus sera affectée au pool concerné. En d’autres termes cet attribut contrôle le passage du workflow par le moment Allocated.

Is Journalizable

Cochée, cette option permet d’historiser toute la structure de l’objet métier concerné par le processus et ce à l’occasion de toute validation d’activité.

Default Task List

Précise le nom de la vue Task List qui s’affiche par défaut dans le cas où l’activité courante du processus n’en spécifie aucune.

Is Exclusive

Cochée, cette option empêche le lancement d’une instance du processus concerné lorsqu’une autre instance (du même processus ou de tout autre processus) est déjà en cours d’exécution sur un même objet métier.

Process Type

Prend l’une des 3 valeurs suivantes :

-      Public : Lorsqu’il s’agit d’un processus pouvant être déclenché par un utilisateur

-      Privé : Lorsqu’il s’agit d’un processus déclenché par un autre processus privé ou public

-      None : Lorsque le type n’est pas spécifié

Is Executable

Si cet attribut est activé, Karaz Builder prendra en compte le processus lors de la génération du PSM associé au modèle. Dans le cas contraire le processus sera considéré comme simple documentation et ne sera pris en compte par Karaz Builder ni par Karaz Workflow Engine.

Is Closed

Indique que le processus ne subira plus de modification.

les propriétés d'une activité utilisateur

Le tableau suivant indique le rôle de chacun des attributs de paramétrage de l’activité :

ATTRIBUT

ROLE

Time to Complete

S’exprime en « dx » ou en « hx » pour indiquer respectivement le nombre de jours ou d’heures nécessaires à la réalisation de l’activité.

View Name

Indique le nom de la vue à laquelle l’activité est associée. Chaque activité de type « User Task » est obligatoirement associée à une vue.

Screen Mask

Indique le nom du masque (usage) qui s’appliquera à la vue, lorsque l’objet sera ouvert pendant l’activité concernée. Chaque activité de type « User Task » doit obligatoirement être associée à un masque. Ce masque sera très utile (Comme nous le verrons à la section usages et masques) pour la gestion dynamique des vues tout au long du cycle d’exécution du processus. Si ce champ n’est pas renseigné, il prendra par défaut le nom de l’activité.

Task List

Indique le nom du bloc dans la vue modèle (model-view.xml) qui correspond à la vue Task List associée à l’activité. Cela correspondra à la vue affichée lorsque l’utilisateur clique sur un objet métier se trouvant au stade de l’activité en question.

Container Xpath

Utilisé uniquement pour les activités de type KMAA. Il n’a pas d’utilité dans le cas des activités normales.

Performer

Indique le rôle (Profil) pouvant exécuter l’activité. Il s’agit d’un élément de documentation uniquement. Les profils étant générés automatiquement pas Karaz pour chaque activité, pour être utilisés librement en post déploiement pour définir les rôles métier des utilisateurs.

Permet de définir le nom et le nom complet (URI : Unified Resource Identifier) d’un service implémentation.

Les activités

Un processus est composé d’un ensemble d’activités reliées par des ressources et des passerelles. Les activités processus peuvent être rangées en 6 types :
  • Activité manuelle (Manual Task)
  • Activité utilisateur (Use Task )
  • Activité automatique (Script Task)
  • Activité règle de gestion (Business Rule Task)
  • Activité service

    • Une activité service se situe à mi-chemin entre l’activité utilisateur et l’activité automatique. Elle permet à un processus métier d’invoquer via message un service tiers qui peut être soit un web service soit un processus tiers. Karaz Workflow Engine attendra le retour du service appelé avant de continuer l’exécution du processus.

    A l’inverse de l’activité automatique, l’activité service peut rester en attente tant que le service appelé n’a pas fourni de retour.

    Les mécanismes de prise en charge des activités service restent cependant assez génériques. Il est donc fortement recommandé d’adopter à la place les activités KMAA introduites plus loin, qui elles proposent des mécanismes optimisés et contrôlés pour réaliser des tâches communément utilisées (Invocation web service, envoi e-mail, …).

  • Activité envoi

    • Une activité de type « envoi » permet d’envoyer un message à un acteur externe. Une fois le message envoyé, l’activité est automatiquement clôturée.

    Comme dans le cas de l’activité « service », l’activité « Envoi » prévue en standard par BPMN 2.0 reste très générique. Pour avoir une utilité réelle et pratique dans l’exécution des processus, Karaz Studio propose une implémentation spécifique sous forme d’activité KMAA
